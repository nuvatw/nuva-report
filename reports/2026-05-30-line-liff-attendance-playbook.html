Codex 小總結

這不是一個 LINE Bot 專案，而是一套打卡狀態系統。

最重要的心智模型是：Flex Message 只做入口，LIFF 做互動介面，後端與資料庫才保存真正的出勤狀態。照 playbook 走，團隊可以從 1 天 health check 推進到可稽核、可補卡、可營運的正式打卡系統。

假設

你口中的「LINE Live」指的是 LINE LIFF（LINE Front-end Framework）。LIFF 是 LINE 內建的網頁應用平台，使用者點 Flex Message、Rich Menu 或 LIFF URL 後，可以在 LINE 裡開啟一個網頁應用，並讓網頁通過 LIFF SDK 取得 LINE 登入狀態、使用者識別資料、好友狀態，或執行部分 LINE 相關動作。

目標

從最小可行專案（MVP）開始，逐步做出一個完整的「個人狀態頁 → 今日區段進度 → 打卡/下班 → Flex Message/Rich Menu 整合 → 後台與生產化」系統。

4.1 為什麼後端要驗證 ID token

前端的 liff.getProfile() 適合拿來顯示頭像和名稱，但不要把它當成可信身份來源。正確做法是：

  1. 前端拿 idToken。
  2. 前端把 idToken 送到你的後端。
  3. 後端呼叫 LINE 的 token verify endpoint。
  4. 後端取得可信的 sub，也就是 LINE user ID。
  5. 後端用 sub 查詢或建立員工資料。

雷點 1：把 LIFF 加在錯誤 Channel

LIFF App 是加在 LINE Login Channel 或 LINE MINI App Channel 下，不是加在 Messaging API Channel 下。

雷點 2：Provider 不同導致 userId 對不上

LINE 官方文件提醒，如果服務放在不同 Provider 下，使用者可能會得到不同 userId。打卡系統最怕身份對不上，所以 Messaging API Channel 和 LINE Login Channel 儘量同 Provider。

雷點 3：Endpoint URL 與實際執行 liff.init() 的 URL 不匹配

如果 Endpoint URL 是 /app/，卻在 / 執行 liff.init()，就可能出問題。最穩做法是 Endpoint URL 設根路徑，或確保所有 LIFF 路由都在 Endpoint 下層。

雷點 4：liff.init() 前修改 URL

LIFF 會在 URL 帶 liff.state、access token 或其他引數。不要在 liff.init() 解析完成前：

  • 清 query string。
  • history.replaceState()。
  • redirect。
  • 把完整 URL 發給 analytics。

雷點 5：Webhook signature 驗證前就 JSON parse

signature 必須用原始 request body 字串/Buffer 計算。任何格式化、轉義、emoji 處理、換行轉換，都可能導致驗證失敗。

雷點 6：用錯 Channel Secret

Webhook signature 使用的是 Messaging API Channel secret，不是 LINE Login Channel secret。

雷點 7：把 access token 或 channel secret 放前端

前端只能有 LIFF ID 和 API base URL。任何 secret 都必須在後端。

雷點 8：以為 Flex Message 會自動更新

Flex 是一張訊息卡片，不會隨著資料庫狀態自動變化。狀態即時顯示要靠 LIFF 頁面或重新發送新 Flex。

雷點 9：沒有處理重複 webhook 和重複打卡

網路重試、使用者連點、LINE webhook redelivery 都很常見。必須有冪等鍵、unique index、event 去重。
